macOS 기반 상용 AI 코딩 에이전트 세션 스토리지 아키텍처 및 상호 운용성(Import/Export) 심층 분석인공지능 코딩 에이전트의 세션 관리 패러다임과 macOS 파일 시스템 아키텍처현대의 인공지능 코딩 에이전트(Claude Code, Gemini CLI, Codex CLI, OpenCode)는 단순한 질의응답을 넘어 로컬 파일 시스템과 상호작용하고, 장기적인 문맥(Context)을 유지하며, 복잡한 소프트웨어 엔지니어링 작업을 자율적으로 수행하는 독립적인 운영 환경으로 진화하였다. 이러한 자율적 에이전트들은 작업의 영속성을 보장하기 위해 대화 기록, 도구 실행 결과, 디버깅 과정, 그리고 모델의 내부 추론(Thoughts) 메타데이터 등을 로컬 머신에 영구적으로 저장한다. 에이전트 세션은 개발자가 작업을 중단하더라도 동일한 문맥에서 즉각적으로 작업을 재개(Resume)할 수 있도록 설계된 핵심적인 상태 관리(State Management) 객체이다.그러나 상용 AI 에이전트 생태계는 세션을 저장하는 위치, 파일의 형식(Schema), 그리고 프로젝트를 식별하는 아키텍처 측면에서 완전히 파편화되어 있다. macOS 환경에서 커맨드라인(CLI) 애플리케이션의 데이터 스토리지는 역사적으로 다양한 표준이 혼재되어 왔다. Apple의 공식 가이드라인은 애플리케이션 지원 파일(Support Files)을 ~/Library/Application Support/에 저장하도록 권장하며 , 사용자 설정을 위해서는 defaults 유틸리티나 ~/Library/Preferences/를 사용할 것을 지시한다. 반면, 수많은 오픈소스 및 유닉스 기반 CLI 도구들은 이식성을 위해 XDG Base Directory Specification을 따라 ~/.config나 ~/.local/share를 선호하거나, 전통적인 유닉스 관습에 따라 홈 디렉토리 하위에 고유의 숨김 폴더(예: ~/.claude, ~/.gemini)를 생성하는 방식을 취한다.이러한 파일 시스템의 파편화와 더불어, 각 에이전트가 프로젝트를 식별하고 세션을 매핑하는 알고리즘(URL 인코딩, SHA-256 해싱, SQLite 관계형 매핑 등)이 상이하다는 점은 work-bridge (github: jaeyoung0509/work-bridge)와 같은 기기 간 세션 임포트/익스포트 오픈소스 프로젝트를 구축하는 데 있어 극복해야 할 가장 큰 기술적 장벽이다. 소스 기기(예: 데스크톱 Mac)에서 생성된 세션을 타겟 기기(예: MacBook)에 완벽하게 '부착(Attach)'하기 위해서는 단순히 파일을 복사하는 것을 넘어, 각 에이전트의 스키마와 경로 결정론(Path Determinism)을 타겟 환경에 맞게 동적으로 재구성하는 고도화된 정규화 및 주입(Injection) 메커니즘이 필수적이다.본 보고서는 macOS 환경을 기준으로 Claude Code, Gemini CLI, Codex CLI, OpenCode의 내부 세션 스토리지 아키텍처를 전면적으로 해부하고, 데이터 형식과 프로젝트 식별 알고리즘을 분석하며, work-bridge가 세션을 정확하게 연동하기 위해 수행해야 할 기술적 마이그레이션 전략을 심층적으로 제시한다.1. Claude Code CLI: 절대 경로 인코딩과 단일 방향 JSONL 아키텍처Anthropic에서 개발한 Claude Code는 프로젝트 간의 컨텍스트 격리성(Project Isolation)을 최우선으로 고려한 세션 스토리지 아키텍처를 보유하고 있다. 중앙 집중식 데이터베이스에 의존하는 대신, 파일 시스템의 디렉토리 구조 자체를 프로젝트 상태 관리자로 활용하는 독특한 접근 방식을 취한다.1.1. 세션 저장소 위치 및 고유 식별자 생성 알고리즘macOS 환경에서 Claude Code는 모든 세션 데이터와 애플리케이션 상태를 사용자의 홈 디렉토리 하위인 ~/.claude/ 경로에 저장한다. 이 디렉토리 내부에는 전역 설정을 담은 settings.json, 전역 시스템 프롬프트인 CLAUDE.md, 그리고 개별 프로젝트의 세션 기록을 담는 projects/ 폴더가 존재한다.Claude Code 아키텍처의 가장 두드러진 특징은 개별 프로젝트를 식별하기 위해 해당 프로젝트의 절대 경로(Absolute Path)를 기반으로 한 URL 인코딩 변환 로직을 사용한다는 점이다. 에이전트가 특정 프로젝트 폴더에서 실행될 때, 시스템은 현재 작업 디렉토리(Current Working Directory)의 절대 경로 문자열에서 영숫자가 아닌 모든 문자를 하이픈(-)으로 치환하거나 URL 인코딩 표준에 맞춰 변환한다.예를 들어, 사용자가 /Users/jaeyoung/work-bridge라는 경로에서 Claude Code를 실행하면, 에이전트는 이를 -Users-jaeyoung-work-bridge라는 문자열로 변환하여 이를 해당 프로젝트의 고유 식별 디렉토리명으로 사용한다. 결과적으로 해당 프로젝트에서 생성된 세션 기록은 다음과 같은 경로에 위치하게 된다.
~/.claude/projects/-Users-jaeyoung-work-bridge/sessions/<session-uuid>.jsonl.이러한 결정론적(Deterministic) 경로 매핑 아키텍처는 에이전트가 별도의 인덱싱 데이터베이스를 조회하지 않고도 현재 위치한 디렉토리를 바탕으로 즉각적으로 세션 폴더를 찾아낼 수 있다는 성능적 이점을 제공한다. 그러나, 이 방식은 기기 간 세션 마이그레이션을 수행할 때 치명적인 제약으로 작용한다. 소스 기기(데스크톱 Mac)와 타겟 기기(MacBook)의 사용자 이름(Username)이 다르거나 상위 디렉토리 구조가 조금이라도 상이할 경우, 타겟 기기의 Claude Code는 복사된 세션 파일을 원본 프로젝트의 것으로 전혀 인식하지 못하게 된다. 또한, 로컬 클라우드 스토리지 동기화 서비스(예: iCloud Drive, OneDrive)를 사용할 경우, macOS의 ~/Library/CloudStorage/ 경로 접근 권한 문제로 인해 데스크톱 앱과 CLI 간의 세션 인식 충돌이 발생하기도 한다.1.2. JSONL 데이터 포맷 및 내부 스키마 세부 구조Claude Code의 세션 파일은 순수하게 추가만 가능한(Append-only) JSON Lines(JSONL) 형식으로 작성된다. 이는 세션 진행 중 커맨드라인 프로세스가 비정상적으로 종료되거나 충돌(Crash)하더라도 데이터 손실을 단일 메시지 수준으로 최소화하기 위한 내결함성(Fault-tolerance) 설계이다.스키마 구성 요소데이터 타입 및 특성내부 동작 메커니즘 및 마이그레이션 시 주의점System MessageJSON Object세션 파일의 첫 번째 레코드에 항상 위치하며, 세션 시작 시 모델에 전달된 완전한 시스템 프롬프트를 포함한다. 도구 정의(Tool definitions), 권한 모드, 주입된 CLAUDE.md 컨텍스트, 그리고 MCP 서버 명령어들이 명시되어 있다.User PromptJSON Object사용자가 터미널을 통해 입력한 자연어 명령어 또는 터미널 내에서 발생한 인터럽트 신호를 기록한다.Assistant ThinkingJSON Object모델의 확장된 추론 과정(thinking)을 포함한다. 이 블록은 디버깅이나 사후 감사 시 모델의 결정 과정을 추적하는 데 핵심적인 역할을 한다.Tool Use & ResultJSON Object모델이 호출한 도구(예: 파일 읽기, Bash 명령어 실행)의 입력값과 그에 대한 출력값을 저장한다. 도구 호출 내역에는 파일의 절대 경로가 빈번하게 하드코딩되므로 주의가 필요하다.Deferred Tools DeltaJSON Object / Metadata모델에게 이미 알려진 도구들의 상태를 추적하는 캐싱 메타데이터이다. 이 데이터가 손상되거나 삭제되면, 세션을 재개할 때마다 모든 도구를 처음부터 다시 선언해야 하므로 프롬프트 캐싱이 무효화되어 심각한 토큰 낭비를 초래하는 버그가 발생할 수 있다.Claude Code는 단순한 텍스트 대화가 아니라 파일 수정 내역과 쉘 스냅샷까지 관리한다. 세션 기록 외에도 ~/.claude/ 하위에는 tasks/ (세션별 태스크 리스트), plans/ (마크다운 기반의 계획 문서), file-history/ (파일 편집 전 체크포인트 해시 백업) 등 다양한 보조 파일들이 함께 생성된다. 또한, 전역적으로 관리되는 ~/.claude/__store.db (SQLite 기반)는 각 세션의 요약 정보(conversation_summaries) 및 전역 프롬프트 히스토리를 인덱싱하는 역할을 수행한다.1.3. work-bridge를 위한 Claude Code 세션 마이그레이션 및 주입 전략Claude Code는 자체적으로 기기 간 세션 임포트/익스포트 명령어를 공식 지원하지 않기 때문에 , work-bridge 프로젝트는 파일 시스템 레벨에서의 정교한 변환 파이프라인을 구축해야 한다. 타겟 머신에서 세션이 "정확하게 잘 붙게" 만들기 위한 핵심 전략은 절대 경로의 완벽한 보정(Mutation)이다.첫째, 타겟 머신의 환경에 맞춘 **경로 재계산(Path Recalculation)**이 필요하다. 소스 기기에서 추출한 <session-uuid>.jsonl 파일을 타겟 기기에 주입할 때, work-bridge는 타겟 기기의 프로젝트 폴더 절대 경로를 파악하여 새로운 URL 인코딩 해시 문자열을 생성해야 한다. 예를 들어, 소스 기기의 경로가 -Users-mac1-project였다면, 이를 타겟 기기에 맞게 -Users-mac2-project로 변경하고, 해당 디렉토리를 ~/.claude/projects/ 하위에 명시적으로 생성한 후 세션 파일을 복사해야 한다.둘째, 내부 페이로드 변환(Payload Mutability) 과정이 수반되어야 한다. JSONL 파일 내부의 각 JSON 레코드, 특히 tool_use와 tool_result 페이로드에는 파일 시스템 탐색, 코드 수정, Bash 스크립트 실행 등과 관련된 절대 경로 문자열이 무수히 하드코딩되어 있다. work-bridge는 이 JSONL 파일을 순회(Iterate)하며 소스 기기의 사용자 홈 경로(/Users/mac1/)를 타겟 기기의 경로(/Users/mac2/)로 일괄 치환(Find and Replace)하는 로직을 수행해야만, 타겟 기기에서 세션 재개 시 컨텍스트 단절이나 파일 읽기 오류(File Not Found)를 방지할 수 있다.셋째, 인덱스 동기화 및 캐시 무효화(Cache Busting) 처리가 필요하다. Claude CLI는 빠른 세션 조회를 위해 sessions-index.json을 사용하거나 __store.db의 데이터를 참조한다. 주입된 세션이 CLI의 /resume 화면에 즉각적으로 노출되도록 하려면, 타겟 기기에 세션을 주입한 후 기존의 sessions-index.json 파일을 업데이트하거나 강제로 삭제하여 CLI가 재실행 시 디렉토리를 다시 스캔하도록 유도해야 한다. 또한 deferred_tools_delta와 같은 캐싱 레코드를 보존하여 주입 후 첫 턴(Turn) 실행 시 프롬프트 캐시가 적중(Hit)하도록 보장하는 것이 비용 효율성 측면에서 필수적이다.2. Google Gemini CLI: 불가역적 해싱과 구조화된 JSON 기반의 폐쇄형 아키텍처Google의 Gemini CLI는 사용자가 터미널을 종료하더라도 작업 컨텍스트를 완벽히 복구할 수 있도록 강력한 백그라운드 자동 저장 메커니즘을 내장하고 있다. 그러나 Claude Code가 텍스트 기반의 경로 인코딩을 사용한 것과 대조적으로, Gemini CLI는 암호학적 해시 알고리즘을 도입하여 시스템 내 프로젝트 상태를 폐쇄적으로 관리한다.2.1. 세션 저장소 위치 및 SHA-256 식별자 알고리즘macOS 환경에서 Gemini CLI의 전역 설정 파일은 ~/.gemini/settings.json에 저장되지만, 모든 프로젝트별 임시 데이터, 액티비티 로그, 그리고 세션 히스토리는 ~/.gemini/tmp/ 디렉토리 하위에 구조화되어 저장된다.에이전트가 특정 프로젝트 디렉토리에서 실행될 때, Gemini CLI는 해당 디렉토리의 절대 경로 문자열을 입력값으로 삼아 SHA-256 해시 함수를 통과시켜 고유 식별자인 <project_hash>를 생성한다.
이렇게 생성된 해시값을 기반으로 각 프로젝트의 세션 저장소 경로가 결정되며, 사용자가 수동으로 체크포인트를 저장하거나 자동 저장된 챗 히스토리는 다음과 같은 경로에 위치하게 된다.
~/.gemini/tmp/<project_hash>/chats/checkpoint-<name>.json 또는 session-<uuid>.json.이러한 SHA-256 해싱 아키텍처는 긴 경로명으로 인한 파일 시스템의 문자열 길이 제한을 회피하고, 디렉토리 간의 충돌을 수학적으로 방지(Collision-resistant)하기 위한 고도화된 설계이다. 또한, 터미널 명령어를 입력하는 쉘 히스토리 역시 프로젝트 해시 단위로 격리되어 ~/.gemini/tmp/<project_hash>/shell_history에 별도 저장되므로, 다른 프로젝트 간 명령어 충돌을 차단한다.그러나 이 불가역적(Irreversible)인 SHA-256 설계는 기기 간 세션 마이그레이션이나 단순한 폴더 이동 시 치명적인 약점을 드러낸다. 단방향 해시 함수의 특성상, ~/.gemini/tmp/에 존재하는 폴더명(해시값)만 보고는 이것이 원래 어떤 절대 경로에서 파생된 것인지 역산출하는 것이 불가능하다. 사용자가 단순히 프로젝트 폴더의 위치를 옮기거나 이름을 변경하기만 해도 절대 경로가 바뀌어 완전히 새로운 SHA-256 해시가 생성되므로, 기존 세션 히스토리에 대한 연결 고리가 끊어져 버린다. 타겟 기기로 세션을 임포트할 때도, 브릿지가 타겟 경로의 SHA-256 해시를 새롭게 계산해내지 못하면 Gemini CLI는 복사된 데이터를 영원히 인식하지 못한다.2.2. 세션 JSON 스키마 및 연관 메타데이터 구조Claude Code나 Codex CLI가 지속적으로 줄을 추가하는 JSON Lines 형식을 선호하는 반면, Gemini CLI는 전체 대화 컨텍스트를 하나의 완성된 구조화된 JSON 객체 트리로 묶어 저장하는 방식을 택하고 있다.스키마/저장소 객체데이터 구조 및 특성브릿지 호환성 및 고려사항Chat History (chats/)JSON Array of Objectscheckpoint-<name>.json 또는 session-<uuid>.json 파일 내부에 messages 배열이 존재하며, 각 객체는 role (user 또는 model)과 parts로 구성된다. 도구 입출력, 토큰 사용 통계, 모델의 추론(Thoughts) 데이터가 모두 이 파일 안에 병합되어 저장된다.Implementation Plans (plans/)Markdown/JSON대규모 프로젝트의 경우 모델이 수립한 다단계 실행 계획(Execution Plan) 문서가 저장된다.Task Trackers (tasks/)JSON현재 진행 중인 세션 내의 구체적인 하위 태스크 성공/실패 여부를 추적하는 상태 객체이다.Tool Outputs (outputs/)Text/JSON터미널 명령어 실행 결과나 파일 시스템 검색 결과 등 크기가 큰 도구 출력값이 별도로 캐싱되는 디렉토리이다.Settings / RetentionJSONsettings.json 내부의 sessionRetention 블록에 의해 기본 30일이 지나면 세션과 연관 데이터가 자동 삭제되는 TTL(Time-To-Live) 정책이 적용된다. 익스포트 전 삭제 방지를 위해 이 설정의 확인이 필요하다.2.3. work-bridge를 위한 Gemini CLI 세션 마이그레이션 및 주입 전략Gemini CLI의 강력한 자동 저장 시스템 덕분에 데이터 유실 위험은 적지만, 불가역적 해시 폴더 구조를 돌파하기 위한 수학적 맵핑 전략이 work-bridge의 핵심이 된다.첫째, **타겟 경로 기반 SHA-256 런타임 계산(Target Hash Generation)**이 수반되어야 한다. work-bridge는 단순히 폴더를 복사하는 것이 아니라, 타겟 머신에서 에이전트를 실행할 위치의 절대 경로 문자열을 읽어들인 뒤, 시스템의 암호화 라이브러리를 사용해 직접 SHA-256 해시를 연산해야 한다. 이후 산출된 <target_project_hash>를 사용하여 타겟 기기의 ~/.gemini/tmp/ 하위에 새로운 해시 디렉토리와 chats/ 폴더를 셋업해야 한다.둘째, JSON 구조 유지 및 페이로드 변환이다. 소스 기기에서 가져온 session-*.json 및 checkpoint-*.json 파일들을 생성된 chats/ 폴더로 이관한다. 이 때, JSON 파일 내부 깊숙이 위치한 parts 객체에 저장된 로컬 파일 경로들이 있다면, 앞선 Claude Code의 사례와 마찬가지로 타겟 환경의 경로 체계로 정규화(Normalization)하는 치환 과정을 거치는 것이 컨텍스트 오류를 예방하는 길이다.셋째, CLI 통합 호출 테스트이다. 데이터 주입이 완료되면, 타겟 기기에서 gemini --resume <UUID> 플래그를 사용하거나 대화형 인터페이스에서 /chat resume <tag>를 입력하여 시스템이 로컬 디렉토리를 스캔하고 주입된 세션 컨텍스트를 성공적으로 메모리에 로드하는지 확인한다. 사용자가 프로젝트 디렉토리를 이동하여 해시가 변경됨으로써 세션이 '삭제된 것처럼' 보이는 현상도, 이와 같은 해시 재계산 및 파일 복사 방식을 통해 복구할 수 있다.3. OpenAI Codex CLI: 글로벌 시계열 구조와 인라인 메타데이터 아키텍처OpenAI의 Codex CLI는 앞서 살펴본 에이전트들이 채택한 '프로젝트별 폴더 격리' 방식과 완전히 궤를 달리하는 아키텍처를 보여준다. Codex CLI는 개별 프로젝트 디렉토리마다 해시 폴더를 생성하지 않고, 시스템 전역(Global)의 중앙 집중형 디렉토리에서 시계열(Time-series) 기반으로 모든 세션 로그를 수집하여 관리한다.3.1. 세션 저장소 위치 및 Date-based 네이밍 규칙macOS 환경에서 Codex CLI는 사용자 세션과 명령 궤적(Trajectory)을 ~/.codex/ 디렉토리 하위에 저장한다. 전역 구성 파일인 config.toml과 모델의 명령 히스토리인 history.jsonl이 루트에 위치하며, 실제 세션의 구체적인 로그들은 날짜별로 그룹화된 sessions/ 폴더에 축적된다.저장되는 세션 파일의 계층 구조는 다음과 같다.
~/.codex/sessions/YYYY/MM/DD/rollout-<timestamp>-<uuid>.jsonl.이러한 설계는 특정 날짜에 수행한 모든 작업 이력을 중앙에서 쉽게 추적하고 디버깅할 수 있다는 장점을 제공한다. 에이전트가 복수의 프로젝트를 횡단하며 수행한 쉘 명령어 내역이나 작업 결과가 단일 타임라인으로 통합되기 때문에, 감사(Auditing) 및 오프라인 분석에 매우 유리한 구조이다. 반면, 수많은 프로젝트의 작업 내역이 동일한 YYYY/MM/DD 폴더에 섞이게 되므로, CLI가 특정 프로젝트의 세션만을 어떻게 선별하여 사용자에게 보여주는지에 대한 고도의 논리적 분리가 필요해진다.3.2. CWD 필터링과 JSONL 이벤트 스키마Codex CLI가 글로벌 저장소에 쌓인 수백 개의 세션 중 현재 사용자가 위치한 프로젝트의 세션만을 필터링하는 방식은 폴더명이 아닌 파일 내부 메타데이터에 전적으로 의존한다.이벤트 스트림 포맷: rollout-*.jsonl 파일은 Claude Code와 유사한 JSON Lines 형식으로, 대화의 각 턴(Turn)과 도구 호출 결과가 순차적 이벤트 스트림 형태로 기록된다. 각 라인은 {timestamp, type, payload} 구조를 가진다.Session Meta와 CWD: 파일의 가장 첫 부분에 기록되는 session_meta 이벤트에는 세션이 시작될 당시의 프로젝트 로컬 절대 경로를 의미하는 cwd (Current Working Directory) 값이 핵심 속성으로 포함되어 있다. 사용자가 터미널에서 codex resume 명령어를 실행하면, 시스템은 ~/.codex/sessions/ 하위의 모든 JSONL 파일을 파싱하여 session_meta의 cwd 값을 읽어들인 뒤, 현재 쉘의 경로와 일치하는 파일만을 찾아내어 이력 선택기(Interactive Picker)에 표시한다.이벤트 타입 요소: 파일 내에는 user_message, agent_message, response_item, 토큰 카운트 이벤트 등 다양한 이벤트 객체들이 순차적으로 저장되며 , 도구 호출의 경우 다중 도구 실행 시 상호 충돌을 방지하기 위한 유니크한 멱등성 키(Idempotency keys)가 부여되어 기록된다.체크포인트 메커니즘: Codex CLI는 메모리 관리를 위해 단순한 대화 압축(Summarization)을 넘어서, 롤아웃 로그를 기반으로 결정론적 프로젝션인 checkpoint_v1.json을 생성할 수 있다. 이는 컨텍스트 윈도우를 리셋하고 시퀀스 뷰를 주입하는 로컬 압축 작업에 사용된다.3.3. work-bridge를 위한 Codex CLI 세션 마이그레이션 및 주입 전략Codex CLI의 아키텍처는 파일 시스템 계층이 단순하여 파일 복사 자체는 용이하지만, 메타데이터 스키마의 일관성을 유지하기 위해 데이터 내부 조작이 필수적인 과제를 안고 있다.첫째, 날짜 기반 계층 구조 복원이다. 소스 기기에서 추출한 rollout-*.jsonl 파일들을 타겟 기기의 ~/.codex/sessions/YYYY/MM/DD/ 경로와 동일한 구조를 생성하여 배치한다.둘째, **CWD 메타데이터 동적 치환(CWD Patching)**이 이 마이그레이션의 가장 중요한 단계이다. 단순히 파일을 복사해 넣는 것만으로는 타겟 기기의 프로젝트 폴더에서 codex resume --last 명령어를 실행했을 때 해당 세션을 전혀 찾을 수 없다. 타겟 기기의 절대 경로가 소스 기기와 다르기 때문이다. work-bridge는 임포트되는 모든 JSONL 파일을 파싱하여 첫 번째 레코드인 session_meta 이벤트를 찾아낸 후, 내부의 cwd 속성값을 타겟 기기의 현재 프로젝트 절대 경로(예: /Users/target_user/my-project)로 정확하게 패치(Patch)해야 한다.셋째, 버전 간 스키마 마이그레이션 및 무결성 검증이다. Codex CLI는 v0.32.0 업데이트 과정에서 rollout JSONL 포맷에 역호환성이 깨지는 대규모 변경(Breaking Change)을 단행한 바 있다. 소스 기기와 타겟 기기의 Codex CLI 버전이 다를 경우, 단순한 경로 치환만으로는 세션 복구 시 에이전트 루프 충돌이 발생할 수 있다. 따라서 브릿지는 JSONL 내부의 type 필드 및 도구 호출의 멱등성 키 배열 구조를 검사하여 타겟 시스템 버전에 부합하는지 호환성을 검증하는 로직을 포함해야 하며, 필요시 압축 보조 파일인 checkpoint_v1.json을 삭제하여 에이전트가 롤아웃 로그로부터 순수하게 상태를 재구성하도록 유도하는 전략이 필요하다.4. OpenCode (Anomalyco): 관계형 데이터베이스(SQLite) 기반 상태 관리 아키텍처OpenCode는 기존 CLI 도구들이 채택한 텍스트 기반 로그(JSONL 또는 단일 JSON 파일) 스토리지 패러다임에서 완전히 탈피하여, 고도화된 RDBMS 아키텍처를 도입한 가장 이질적이고 복잡한 시스템이다. 복잡한 멀티 에이전트 워크플로우와 계층적인 서브 에이전트(Sub-agent) 태스크 트리를 관리하기 위해 SQLite 단일 데이터베이스를 활용한다.4.1. 세션 저장소 위치 및 Drizzle ORM 마이그레이션 이력초기 버전의 OpenCode는 macOS에서 ~/.local/share/opencode/storage/session/{projectID}/{sessionID}.json 형태의 파일 기반 저장소를 사용하였다. 그러나 파일 시스템 파편화 문제와 디렉토리 이동 시 컨텍스트 상실 문제를 해결하기 위해, v1.0 후반부부터 Drizzle ORM 엔진을 도입하여 SQLite 기반으로 데이터 레이어를 전면 재작성하였다.현재 macOS 기준 OpenCode의 모든 글로벌 프로젝트 정보, 워크스페이스 환경, 세션 히스토리 및 챗 메시지는 단 하나의 SQLite 파일인 ~/.local/share/opencode/opencode.db 내부에 관계형으로 기록된다 (로컬 개발 빌드나 특정 릴리즈 채널의 경우 opencode-local.db 등의 파생된 파일명을 가지기도 한다).4.2. 관계형 스키마(RDBMS) 구조 및 데이터 블로트(Bloat) 문제OpenCode의 SQLite 데이터베이스는 다수의 테이블이 외래 키(Foreign Key) 및 캐스케이드 삭제(ON DELETE CASCADE) 제약조건으로 견고하게 얽혀 있는 구조를 지닌다.SQLite 테이블 이름데이터 스키마 특성 및 저장 내용마이그레이션 시 기술적 고려사항project로컬 머신 내의 개별 프로젝트 디렉토리 경로 정보 및 프로젝트 식별자(id)를 관리한다.기기 간 마이그레이션 시, 이 테이블에 하드코딩된 소스 기기의 절대 경로를 타겟 기기의 경로로 UPDATE 해야 한다.workspace특정 브랜치 상황 및 환경 설정(Config)에 결속된 워크스페이스를 정의하며, project_id를 참조한다.Drizzle ORM의 마이그레이션 버그(v1.2.16)로 인해 이 테이블 생성이 누락되는 심각한 스키마 손상 사례가 보고되었다.session사용자가 실행한 개별 대화 세션과 모델 정보를 정의하며, workspace_id를 참조하여 자신이 어떤 프로젝트와 환경에 속하는지 증명한다.전체 세션 중 약 86%가 메인 세션이 아닌 task() 함수 호출로 생성된 자식(Child) 세션일 정도로 계층 구조가 복잡하다.message각 세션에서 발생한 개별 턴(Turn) 단위의 사용자/어시스턴트 메시지 메타데이터이다.session_id와 강하게 결합되어 있다.part실제 메시지 내부의 텍스트 청크, 코드 블록, 도구 호출 파라미터, 그리고 도구 실행 출력 로그(Tool outputs)를 분할하여 저장한다.전체 DB 용량의 81% 이상을 차지할 수 있는 데이터 팽창(Bloat)의 주범이며, 마이그레이션 시 이 테이블의 거대한 페이로드를 이관하는 비용을 고려해야 한다.이러한 관계형 아키텍처는 쿼리 성능 면에서 압도적인 장점을 지니지만, 기기 간 이관 시 데이터 무결성을 보장하기 어렵다는 취약점을 낳는다. 특히, .git 디렉토리가 존재하는 프로젝트 폴더 내부에서 CLI를 실행할 경우, 전역(Global) 세션 리스트가 차폐되는 버그가 존재하는 등 로컬 파일 시스템 탐색과 DB 상태 간의 동기화 불일치 이슈가 존재한다.4.3. work-bridge를 위한 OpenCode 세션 마이그레이션 및 주입 전략OpenCode의 경우, JSONL 파일 시스템 구조에 직접 개입하던 타 에이전트 방식(Raw Injection)을 동일하게 적용하는 것은 매우 위험하다. SQLite의 __drizzle_migrations 테이블 타임스탬프 불일치, DB 락(Lock), 외래 키 무결성 위반 등의 심각한 손상을 유발할 수 있기 때문이다. 따라서 work-bridge는 다음과 같이 안전하게 설계된 어플리케이션 계층(Application Layer) 인터페이스를 우회적으로 활용해야 한다.첫째, 공식 CLI Export/Import 프로토콜 활용이다. OpenCode 생태계는 타 에이전트에 비해 외부 통합 요구를 수용하여, 구조적으로 opencode session export -f json 과 같은 커맨드 라인 인수를 확장하고 있다. 브릿지 애플리케이션은 소스 기기에서 백그라운드 쉘 프로세스로 이 공식 명령어를 호출하여 세션 데이터를 정형화된 JSON 포맷으로 추출해야 하며, 타겟 기기에서도 제공되는 임포트 파이프라인을 트리거하여 SQLite 엔진이 스스로 트랜잭션을 관리하며 데이터를 적재하도록 위임(Delegate)해야 한다.둘째, 로컬 HTTP 서버 기반 API 브릿징 전략이다. OpenCode는 CLI 뿐만 아니라 백그라운드 데몬 및 웹 UI 통신을 위한 로컬 서버 구조를 채택하고 있다. 타겟 머신의 ~/.local/share/opencode/opencode.db 파일을 직접 수정하여 프로젝트 절대 경로를 UPDATE 하는 위험한 쿼리(Raw Query) 실행을 피하기 위해, work-bridge는 타겟 기기의 OpenCode 로컬 서버로 POST /session/:sessionID/message 등의 REST API 호출을 통해 데이터를 프로그래밍 방식으로 주입(Replay)하는 방식을 고려해야 한다.셋째, 하위 호환성 폴백(Fallback) 관리이다. 만약 사용자가 여전히 구버전의 JSON 기반 스토리지(~/.local/share/opencode/storage/session/)를 사용하는 환경에 머물러 있다면 , 브릿지는 이를 감지하여 SQLite 주입 방식을 중단하고 기존 방식대로 파일 기반 복사 및 경로 치환(URL 디코딩/인코딩) 로직으로 폴백(Fallback)해야 한다. 또한 .git 컨텍스트 충돌 버그를 우회하기 위해, 세션 주입 시에는 프로젝트 폴더 바깥의 임시 작업 영역에서 브릿지 연산을 수행한 뒤, SQLite 내부의 project 테이블 경로 참조만을 동기화하는 안정화 단계를 거쳐야 한다.5. work-bridge (오픈소스) 통합 설계 및 범용 상호 운용성 구현 지침상용 AI 에이전트 간의 세션 상태는 단순한 텍스트 로그가 아닌, 파일 시스템, 암호학적 해싱, 데이터베이스 스키마, 메타데이터 상태 머신이 복합적으로 얽혀 있는 운영 체제의 확장에 가깝다. GitHub jaeyoung0509/work-bridge 프로젝트가 Claude Code, Gemini CLI, Codex CLI, OpenCode 간의 세션 데이터를 macOS 시스템 위에서 완벽하게 "붙이기" 위해서는, 각기 다른 스토리지 철학을 중앙에서 조율하는 **정규화 계층(Normalization Layer)**과 **경로 변환 엔진(Path Translation Engine)**을 통합 구축해야 한다.5.1. 정규화된 중간 표현(Canonical IR) 스키마 설계각 에이전트의 포맷(JSONL, 단일 JSON, SQLite 테이블)을 1:1로 다중 변환하는 N x N 매트릭스 방식은 스키마 변경 시 유지보수를 불가능하게 만든다. 오픈소스 프로젝트 cross_agent_session_resumer의 아키텍처 사례를 참조하여 , 추출된 모든 에이전트의 세션 데이터를 독립적인 통합 인메모리 포맷(Canonical Intermediate Representation, IR)으로 먼저 파싱(Parsing)하는 중앙 파이프라인을 설계해야 한다.Read (Source): 소스 에이전트(예: Gemini CLI)의 checkpoint-*.json 페이로드를 읽어들여 공통 IR로 매핑한다. 이 IR은 {id, timestamp, project_cwd, messages[{role, type, text, tool_calls, tool_results}], metadata}의 일관된 구조를 가진다.Transform (Mutation): IR 상태에서 절대 경로 변환 엔진이 개입하여 소스 기기의 환경 의존성을 타겟 기기의 환경으로 교체한다.Write (Target): 타겟 에이전트(예: Codex CLI)의 포맷 컨버터가 활성화되어, IR 데이터를 다시 rollout-*.jsonl 형식으로 직렬화(Serialize)하여 타겟 파일 시스템 구조에 기록한다.5.2. 절대 경로 보정 엔진 (Absolute Path Mutator)기기 간, 혹은 툴 간 세션 부착(Attach)이 실패하는 가장 빈번한 원인은 소스와 타겟의 머신 환경 차이(예: /Users/alice/projects/ vs /Users/bob/workspace/) 때문이다. work-bridge의 'Path Translation Engine'은 페이로드 내부의 문자열을 정규표현식이나 AST 파싱을 통해 스캔하고 다음의 보정 로직을 자동화해야 한다.해시 및 디렉토리 재생성:Claude Code 호환 타겟팅: 타겟 기기의 프로젝트 절대 경로를 바탕으로 하이픈 치환 및 URL 인코딩을 수행하여 새로운 폴더(~/.claude/projects/<encoded_hash>)를 생성한다.Gemini CLI 호환 타겟팅: 타겟 프로젝트 절대 경로를 가져와 SHA-256
